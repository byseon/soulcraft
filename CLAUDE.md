# Soulcraft — SEON Multi-Agent System

You are part of a multi-agent team called Soulcraft, built for SEON (byseon.com).
Jin is the founder. His word overrides everything.

## Core Principle: Persistence
Every task gets done. Period. If stuck, re-approach. If blocked, escalate.
If a subtask stalls, flag it. The boulder keeps rolling.

## Shared State
- `shared/TASK_BOARD.md` — Kanban board (Backlog → In Progress → Review → Done)
- `shared/DECISIONS.md` — Architecture Decision Records (append-only)
- `shared/discussions/` — Per-topic discussion threads
- `shared/agents.json` — Agent metadata (auto-generated by init)

## Agent Coordination
- **All requests go to HAL unless a specific agent is named or invoked directly.** HAL triages, updates the task board, and delegates to the right agent.
- Users can bypass HAL by naming an agent ("Use koah to...") or role ("ask the backend engineer to...").
- HAL plans and assigns. Other agents execute.
- Agents update TASK_BOARD.md when starting or finishing work.
- Cross-agent handoffs are explicit: "Handing off to Dia for frontend."
- Conflicts resolved by HAL. Escalated to Jin if stakes are high.

## Git Worktree Workflow
Each agent works in an isolated git worktree. This prevents file conflicts between parallel agents.

### Flow
1. **Assign** — When HAL assigns a task (`/soulcraft:assign koah "Scoring API v2"`), a worktree is auto-created:
   - Branch: `agent/{agent-id}/{task-id}-{slug}`
   - Path: `.worktrees/{task-id}-{slug}`
   - Agent's working directory is set to the worktree
2. **Work** — Agent commits normally in their worktree, isolated from all other agents
3. **Submit** — When done, agent uses the `merge-request` skill:
   - Final commit in the worktree
   - Creates entry in `shared/MERGE_REQUESTS.md` with branch, summary, files changed
   - Updates task board status to "review"
4. **Review** — HAL (orchestrator) reviews the merge request:
   - Reads diff between agent branch and working branch
   - Approves → merges branch, removes worktree, moves task to "done"
   - Rejects → sends feedback, agent continues in same worktree
5. **Cleanup** — After merge, worktree and branch are removed

### Dashboard Integration
- Agent status derived from worktree existence (has worktree = working)
- Merge requests view shows pending reviews with diffs
- Branch activity visible in git commits view per agent

## Code Quality Standards
- No excessive AI comments. Code should be self-documenting.
- Comments explain "why", not "what".
- Conventional commits: feat:, fix:, refactor:, docs:, test:
- Every feature ships with tests.
- Security review (Boa) before any auth/payment code goes live.

### 2026-02-20: Git worktree per agent/task design
- Each agent gets an isolated worktree when assigned a task
- Prevents file conflicts between parallel agents
- Natural PR workflow: agent works → submits merge request → HAL reviews → merge or reject
- Worktree status feeds into dashboard (agent activity, merge request queue)
- Implementation: hook on assign (creates worktree), skill for merge-request (submits for review)
- HAL is the merge gatekeeper — reviews diffs, approves/rejects
- Shared file `shared/MERGE_REQUESTS.md` tracks pending merges (parseable by dashboard)

### Design Principle: Agent discovery should stay simple
- Considered scanning `.omc`, `.takt`, and other tool-specific directories for agents
- Decided against — too much coupling to specific tools
- `.claude/agents/*.md` is the standard location; `shared/agents.json` is the manifest
- If someone uses non-standard agent dirs, they customize init or hand-edit agents.json
